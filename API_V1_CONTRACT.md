# Cyber32 API v1 Contract

This document defines the official Cyber32 v1 API contract for ESP32.

Cyber32 v1 is capability-first.

The API is the only supported external interface for:

- Dashboard
- Mobile Studio, future v2+
- external applications
- local integrations

This contract follows:

- `ARCHITECTURE.md`
- `CAPABILITY_SCHEMA.md`
- `CAPABILITY_PAYLOAD_SCHEMA.md`
- `COMMAND_DISPATCH_CONTRACT.md`
- `ERROR_MODEL.md`
- `REGISTRY_IMPLEMENTATION_PLAN.md`
- `ESP32_V1_SCOPE.md`
- `MOBILE_STUDIO_VISION.md`

## 1. API Principles

### Capability-First

All API interactions with system behavior must use capability IDs.

Correct:

```text
GET /capabilities/CAP_TEMPERATURE
POST /capabilities/CAP_SERVO_POSITION/command
```

Incorrect:

```text
POST /devices/servo-1/write
POST /modules/gps/start
```

### Machine-Readable

API responses must be machine-readable.

API must preserve:

- `capability_id`
- `schema_version`
- canonical units
- machine-readable values
- `error_code`
- `request_id`

Friendly display names are optional metadata only.

### Bounded Responses

Cyber32 v1 runs on ESP32. API responses must be bounded.

API must not return:

- large metadata blobs
- full logs
- full telemetry history
- Dashboard HTML in JSON responses
- unbounded arrays
- raw internal memory state

### ESP32-Friendly

API must be small, predictable, and non-blocking where possible.

Rules:

- keep request bodies small
- keep responses small
- rate-limit polling
- reject oversized requests
- avoid dynamic allocation
- avoid long blocking work
- serialize only needed fields

### No Module-Name Dependency

API may expose module names as display metadata.

API commands and Logic-facing data must not depend on module names.

Capability IDs are canonical.

## 2. Authentication Model

### v1 Local Mode

Cyber32 v1 assumes local ESP32 operation.

Default v1 mode:

```text
local network or ESP32 access point
single user/operator assumption
minimal API surface
```

Even in local mode, API must not expose raw Device, Driver, or HAL control.

### Optional Auth

Optional v1 authentication may include:

```text
static token
local password
setup-mode-only access
```

If auth is enabled:

- unauthorized requests return `ERR_API_UNAUTHORIZED`
- command endpoints require authorization
- read-only endpoints may be optionally public in setup mode

### Future Expansion

Future v2+ may add:

- user accounts
- roles
- signed requests
- app pairing
- encrypted transport
- cloud authentication

These are not required for ESP32 v1.

## 4. Request Schema

All requests should be compact.

Common request fields when a body is used:

```text
schema_version
request_id
capability_id
```

### Required Fields

For command requests:

```text
schema_version
request_id
command
params
timeout_ms
```

`capability_id` is supplied by the URL path:

```text
POST /capabilities/{id}/command
```

The body may repeat `capability_id` only for validation. If body and path disagree, the request must fail.

### schema_version

Required for body payloads.

v1 value:

```text
1
```

### request_id

Required for command requests.

Rules:

- maximum 16 bytes in ESP32 v1
- stable for request tracking
- generated by client or API if omitted only when documented

### capability_id

Rules:

- must start with `CAP_`
- must match `CAPABILITY_SCHEMA.md`
- must exist in Registry for state or command operations
- must not be a module name

## 5. Response Schema

### Success Response

Common success response:

```text
ok
schema_version
request_id
data
timestamp_ms
```

Example:

```text
ok: true
schema_version: 1
request_id: req-001
timestamp_ms: 12345
data: <endpoint-specific object>
```

### Error Response

Common error response:

```text
ok
schema_version
request_id
error
timestamp_ms
```

Error object:

```text
error_code
severity
recoverable
target_id
safe_action
message_key
```

Example:

```text
ok: false
schema_version: 1
request_id: req-001
timestamp_ms: 12345
error:
  error_code: ERR_CAPABILITY_UNAVAILABLE
  severity: error
  recoverable: true
  target_id: capability:CAP_DISTANCE
  safe_action: disable_capability
  message_key: capability_unavailable
```

### Command Response

Command response:

```text
ok
schema_version
request_id
capability_id
command_state
accepted
error
timestamp_ms
```

Command states:

```text
request
accepted
executing
completed
failed
timed_out
cancelled
```

`accepted: true` means the Service accepted the command for execution. It does not always mean execution completed.

## 6. Error Handling

API must use canonical error codes only.

Common API errors:

```text
ERR_API_UNAUTHORIZED
ERR_CAPABILITY_UNAVAILABLE
ERR_CONFIG_INVALID
ERR_REGISTRY_FULL
ERR_RUNTIME_OVERLOAD
ERR_DEVICE_TIMEOUT
ERR_STORAGE_WRITE_FAILED
```

Rules:

- API must expose machine-readable `error_code`.
- API must not translate `error_code`.
- API may include `message_key` for Dashboard/Mobile Studio translation.
- API must not expose stack traces.
- API must not expose large logs by default.
- Unauthorized command requests return `ERR_API_UNAUTHORIZED`.
- Unknown capability IDs return a canonical capability or registry error.

## 7. Capability State Responses

Capability state responses use `CAPABILITY_PAYLOAD_SCHEMA.md`.

`GET /capabilities/{id}/state` returns the latest Registry payload state.

Common fields:

```text
capability_id
schema_version
timestamp_ms
available
stale
value
unit
quality
error_code
```

Rules:

- Registry remains source of truth.
- API does not own payload state.
- API serializes Registry state.
- Unavailable values must not be represented as zero.
- Stale values must include `stale: true`.
- Friendly names may be included separately as display metadata.

## 8. Command Responses

Command requests use `COMMAND_DISPATCH_CONTRACT.md`.

Command flow:

```text
API
-> Service
-> Runtime
-> Device
-> Driver
-> HAL
```

Rules:

- API never calls Devices directly.
- API never calls Drivers directly.
- API never calls HAL directly.
- API never bypasses Services.
- API never owns command state.
- Registry stores latest command state.
- Events announce command state changes.

## 9. Event Responses

`GET /events` exposes recent compact events from the bounded event model.

Event response fields:

```text
event_id
category
priority
timestamp_ms
source_layer
source_id
target_id
status
value_type
value
```

Rules:

- API must not expose unbounded event history.
- API must not expose large payloads in events.
- API may expose only recent events or counters in v1.
- Event Bus is not Registry.
- Consumers needing current state must query Registry endpoints.

## 10. ESP32 Limits

Request and response limits:

```text
maximum request body:       1024 bytes
maximum response body:      4096 bytes
maximum request_id length:  16 bytes
maximum capability_id length: 32 bytes
maximum command params string: 64 bytes
simultaneous API clients:   2
WebSocket clients:          0 by default, 1 if enabled later
```

Rate limits:

```text
GET /system/status:              <= 2 requests per second
GET /capabilities:               <= 1 request per second
GET /capabilities/{id}/state:    <= 5 requests per second per client
POST /capabilities/{id}/command: capability-specific
GET /events:                     <= 2 requests per second
GET /errors:                     <= 1 request per second
GET /registry/summary:           <= 1 request per second
```

Polling recommendations:

- Dashboard should poll `/system/status` slowly.
- Dashboard should poll specific capability states instead of all state when possible.
- Mobile Studio future app should cache display metadata.
- Commands should not be spammed.
- Safety stop commands should remain high priority.

ESP32 rules:

- reject oversized requests
- prefer compact JSON
- avoid large arrays
- avoid dynamic allocation
- avoid long request handlers
- do not block Runtime safety work

## Endpoint Contracts

### GET /system/status

Returns current system status.

Data fields:

```text
system_state
runtime_state
safe_mode
degraded
uptime_ms
free_heap
capability_count
module_count
device_count
service_count
latest_error_code
```

### GET /system/version

Returns version information.

Data fields:

```text
api_version
system_version
schema_version
board
framework
build_profile
```

### GET /capabilities

Returns compact capability list.

Each item:

```text
capability_id
name
category
kind
access
available
stale
unit
```

The list must be bounded by `CYBER32_MAX_CAPABILITIES`.

### GET /capabilities/{id}

Returns capability metadata and compact state summary.

Data fields:

```text
capability_id
name
description
category
kind
data_type
access
source
available
stale
owner_ref
```

`owner_ref` may be compact and should not be used by Logic as a module-name dependency.

### GET /capabilities/{id}/state

Returns latest capability payload state.

Data follows `CAPABILITY_PAYLOAD_SCHEMA.md`.

### POST /capabilities/{id}/command

Requests command execution through Services.

Body:

```text
schema_version
request_id
command
params
timeout_ms
```

Response follows command response schema.

### GET /events

Returns recent compact events or event counters.

Data fields:

```text
events
overflow_count
dropped_low_count
coalesced_count
```

`events` must be bounded.

### GET /errors

Returns latest compact errors.

Data fields:

```text
errors
count
```

Each error:

```text
error_code
severity
recoverable
timestamp_ms
source_layer
target_id
state
safe_action
message_key
```

### GET /registry/summary

Returns compact Registry summary.

Data fields:

```text
modules_used
modules_max
devices_used
devices_max
capabilities_used
capabilities_max
services_used
services_max
errors_used
errors_max
metadata_refs_used
metadata_refs_max
```

## Dashboard And Mobile Studio Rules

Dashboard and future Mobile Studio must use API only.

They may show:

- friendly names
- icons
- translations
- display units
- user-friendly errors

They must preserve:

- `CAP_*` IDs
- canonical payload fields
- canonical units
- machine-readable `error_code`
- command `request_id`

They must not:

- access Devices directly
- bypass Services
- depend on module names
- treat friendly labels as rule identity

## Boundary Rules

- API never calls Devices directly.
- API never bypasses Services.
- API never owns state.
- Registry remains source of truth.
- Dashboard and Mobile Studio use API only.
- Capability IDs remain canonical.
- Friendly names are optional display metadata.

## Success Criteria

Cyber32 API v1 is successful when:

1. All external integrations use API only.
2. API requests use capability IDs.
3. Command requests flow through Services.
4. API responses are bounded for ESP32.
5. API exposes canonical error codes.
6. API serializes Registry state without owning it.
7. Dashboard and future Mobile Studio can build friendly UI while preserving `CAP_*` IDs.
